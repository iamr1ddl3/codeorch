Haiku-verdict-only matches Haiku-weighted on score (+0.012 delta) at 3.9× the speed and 44% the cost. This invalidates the Day 5 documented variance: there's no defensibility benefit from the component breakdown that justifies the ~5× latency tax. Recommend swapping default Gate to verdict-only and storing the rubric breakdown only on retry/escalate paths where decomposition aids debugging.
Sonnet-verdict scores +0.034 higher than Haiku-verdict at 3.5× the cost. On easy tasks (the dominant data here), the score premium does not justify the price. Consider Sonnet-verdict as a 'second opinion' on Haiku-verdict scores below 0.75 — buys judgment only when the cheap gate is uncertain.
Sonnet-verdict had 1 JSON-format error(s) — the model occasionally returns prose analysis instead of structured JSON when given a verdict-only prompt. Production deployment of any Sonnet variant needs response-format enforcement (tool-use mode or strict JSON mode) before considering it stable.

Method

Each of 13 prior CodeOrch runs (with successful plan + code + tests stages persisted in pgvector) was replayed through 3 gate variants — same inputs, different scoring strategy. This isolates the gate decision from upstream agent variance.

  • haiku-weighted — Haiku 4.5 + 4-component weighted rubric output (CodeOrch production default through Day 6).
  • haiku-verdict — Haiku 4.5 + verdict-only output (score + verdict + issues, no decomposition).
  • sonnet-verdict — Sonnet 4.6 + verdict-only output (more capable model, simpler prompt).

Cost computed from token usage at public Anthropic pricing (Haiku: $0.80/$4.00 per M in/out tokens; Sonnet: $3/$15 per M in/out). Latency is per-call LLM time, captured from the SDK return — excludes DB write + Langfuse flush.

Why this matters for Rocket

Dhruv Gandhi's exact framing: "RL signals adjust routing weights when model-task combos underperform." RouterBench is the empirical version — before deciding what model to route to, you measure what each model costs in dollars and seconds for a given task, and you change the routing table when the cheaper option is statistically indistinguishable on quality. That's the case here: production should switch the default Gate to haiku-verdict-only, with the weighted rubric kept as a debug-mode-only path on retry / escalate verdicts.
